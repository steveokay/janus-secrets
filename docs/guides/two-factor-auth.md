# Two-factor authentication (TOTP)

**Two-factor authentication (2FA)** adds a second credential to a password
login: after your email + password, Janus asks for a short **time-based
one-time code (TOTP)** generated by an authenticator app on your phone. A
stolen or guessed password is no longer enough on its own — an attacker would
also need the rotating code, which never leaves your device.

Janus implements 2FA with **TOTP** (the RFC 6238 standard used by Google
Authenticator, 1Password, Authy, and every other authenticator app) plus a set
of single-use **recovery codes** for the day you lose your phone. Enrolment is
self-service — you turn it on from the web UI — and once it is active, every
password login (web *and* CLI) requires a code.

This guide covers enabling 2FA, signing in with it, recovery codes, and turning
it off. For managing your active sessions, see the Settings → Active sessions
panel and `janus session`; for the master-key / seal operations that also live
in Settings, see [master-key-and-backup.md](./master-key-and-backup.md).

## What 2FA protects — and what it doesn't

2FA guards **password logins** — the UI login and `janus login`. It is the
widest door into the vault, so it is the one most worth a second lock.

- **Service tokens** have no second factor by design — they are machine
  credentials, scoped and revocable, and are meant to be stored in a secret
  manager rather than typed by a person. See
  [service-tokens.md](./service-tokens.md).
- **OIDC / SSO logins** are not gated by Janus's TOTP: when you sign in through
  an identity provider, that provider owns your multi-factor policy. Enforce MFA
  at the IdP for those accounts.

If your account can log in with *both* a password and OIDC, enabling TOTP
protects only the password path; make sure the IdP enforces MFA too.

## Enabling 2FA

Enrolment happens in the web UI, under **Settings → Two-factor authentication**.

1. Click **Enable two-factor**. Janus generates a fresh secret and shows it to
   you **once**, as a QR code plus the raw secret and an `otpauth://` setup
   link. The secret is rendered in your browser and never leaves it.
2. In your authenticator app, **scan the QR code** — or, if you can't scan,
   paste the setup link or type the secret in manually.
3. Your app now shows a 6-digit code that changes every 30 seconds. Enter the
   current code into the **Verification code** field and submit.
4. On success, Janus activates the factor and shows your **recovery codes**
   once (see below). Save them, then dismiss the panel.

Until you complete step 3, the factor is only *pending* — it does not gate your
login, and starting enrolment again replaces the pending secret. You cannot
begin a second enrolment while 2FA is already active; disable it first.

### What the app is storing

The QR encodes a standard `otpauth://totp/…` URI: issuer `Janus`, your account
label, the shared secret, and `SHA1 / 6 digits / 30-second period` — the
interoperable TOTP defaults, so any RFC 6238 app works. On the server, the
160-bit secret is encrypted under the instance master key (bound to your user
id) and is re-wrapped automatically if the master key is ever rotated.

## Signing in with 2FA

Once 2FA is active, the code is required on every password login.

### Web

Enter your email and password as usual. When Janus recognises an activated
factor it reveals a **Two-factor code** field; enter the current code from your
app (or a recovery code) and submit.

### CLI

`janus login` prompts for the code when the server asks for it:

```sh
$ janus login
Email: you@example.com
Password:
Two-factor code: 123456
# → signed in
```

The client sends `totp_code` alongside your credentials and retries the login
transparently. You can also supply it up front by answering the prompt from a
script's stdin if you automate `janus login` (though for automation a service
token is usually the better fit — see [service-tokens.md](./service-tokens.md)).

### What the login endpoint does

`POST /v1/auth/login` accepts an optional `totp_code`. If your account has an
activated factor and you omit the code, the endpoint returns
`401 { "error": { "code": "totp_required" } }` — a signal to collect the code,
not a failed login. A wrong code returns the same generic invalid-credentials
error as a wrong password, so the endpoint never reveals whether a given account
has 2FA enabled to someone who doesn't already hold the password.

## Recovery codes

When you activate 2FA (and whenever you regenerate them) Janus issues a set of
**single-use recovery codes** and shows them exactly once. Each one can stand in
for a live authenticator code at login — this is how you get back in if you lose
your phone.

- **Save them somewhere safe and offline** — a password manager entry, a printed
  copy in a drawer. They are shown once and never again.
- **Each code works once.** Using one at login consumes it. Settings shows how
  many remain and warns when you are running low.
- **Regenerate** from Settings → Two-factor authentication (**Regenerate
  recovery codes**) — you confirm with a current code, and the whole previous
  set is invalidated and replaced. Regenerate if you have used most of them or
  suspect a copy leaked.

On the server, recovery codes are never stored in the clear: Janus keeps only an
HMAC-SHA256 of each (normalised, so dashes and case don't matter), so a database
dump cannot be used to derive them, and a used code cannot be replayed.

## Disabling 2FA

From **Settings → Two-factor authentication**, choose **Disable two-factor**.
You confirm with a current authenticator code *or* a recovery code; on success
the factor and all its recovery codes are removed, and password login stops
asking for a second factor.

Disabling then re-enabling issues a brand-new secret and a brand-new set of
recovery codes — the old secret and old codes are gone and cannot be reused.

## API reference

All routes require a **user session** (service tokens are rejected) and are
rate-limited. Codes, secrets, and the `otpauth` URI never appear in the audit
log; enrolment, confirmation, and disabling each write a value-free audit event.

| Method & path | Purpose |
| --- | --- |
| `GET /v1/auth/totp` | Status: `{ enabled, recovery_remaining }`. |
| `POST /v1/auth/totp/enroll` | Begin enrolment; returns `{ secret, otpauth_url }` (shown once). |
| `POST /v1/auth/totp/confirm` | Body `{ code }`; activates and returns `{ recovery_codes }` (shown once). |
| `POST /v1/auth/totp/disable` | Body `{ code }` (TOTP or recovery code); removes the factor. |
| `POST /v1/auth/totp/recovery-codes` | Body `{ code }`; regenerates and returns a fresh `{ recovery_codes }`. |

## Security notes

- **Codes and secrets are never logged or audited.** The audit log records that
  you enrolled / confirmed / disabled 2FA, never the values involved.
- **The secret stays wrapped at rest** under the master key and is bound to your
  user id, so a secret row cannot be transplanted onto another account.
- **Verification is constant-time** and allows a ±30-second window for clock
  drift, so a correct code is accepted even if your phone and the server differ
  by a few seconds.
- **Losing both your phone and your recovery codes** means an administrator must
  intervene: an admin can disable the account (`POST /v1/users/{id}/disable`)
  and re-provision it. Keep at least one recovery code somewhere you can reach
  without your phone.
