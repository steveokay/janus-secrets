# Nocturne N1 — Tokens, Shell, Motion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land phase N1 of the Nocturne UI redo — a new `theme.css` token system (dark-first + light derivation), the canvas/elevation/glow/motion foundation, a restyled shell + command palette + kit primitives, a new canonical mockup, and the CLAUDE.md design-authority handover — as one gated PR that changes **look only**, not behavior or security posture.

**Architecture:** Nocturne is applied primarily by **redefining the values of existing CSS-variable tokens** in `web/src/theme.css` (so every component that already consumes token classes restyles for free) plus **adding new tokens/utilities** (translucent surface layers, elevation, violet glow, a 5-step ink scale, a gradient canvas, and a real motion system) surfaced through `tailwind.config.js`. Component `.tsx` files change only where a treatment can't be expressed as a pure token swap (gradient nav-active wash + inset rail, glow on the logo/primary button, backdrop-blur modals). The two guard tests (`no-raw-palette`, `dark-aa`) and the dual-theme `smoke` are the primary correctness gate for a restyle; behavioral tests stay green untouched.

**Tech Stack:** React + TypeScript + Vite + Tailwind (CSS-variable tokens) + Vitest/RTL/MSW + a puppeteer smoke.

**Source of truth:** `ui-redo.md` §3 (design language, exact values), §4.9 (shell), §5 (N1 scope), §7 (authority handover). Reference renders: `.superpowers/brainstorm/1307-1783872022/content/visual-direction.html` (card A = Nocturne) and `nocturne-home.html` (home). Exact values below are transcribed from those; if a value here disagrees with `ui-redo.md` §3, §3 wins.

## Invariants (re-verify after every task — from ui-redo.md §1)

- **Only `web/src/theme.css` may contain hex color literals.** Everything else uses token classes or `rgba()`/`var()`. Enforced by `web/src/test/no-raw-palette.test.ts` (scans all `.ts/.tsx/.css` under `web/src` except itself and `theme.css`). New utility CSS must use `var()`/`rgba()`, never hex.

## Task 2 — Canvas + real motion (global layer)

**Files:**
- Modify: `web/src/index.css` (confirm the actual global stylesheet path first: `grep -rl "prefers-reduced-motion\|@tailwind" web/src`)

**Gotcha (must not break smoke):** `smoke.mjs` reads `getComputedStyle(document.documentElement).backgroundColor` and requires the dark sum < 120. A gradient applied via `background-image` leaves `background-color` transparent → smoke fails. So set **both**: a solid `background-color: var(--canvas-base)` (what smoke reads) **and** `background-image: var(--canvas)` (the visible gradient) on `html`.

- [ ] **Step 1: Find the global stylesheet.**

Run: `grep -rln "@tailwind\|prefers-reduced-motion\|:root" web/src --include=*.css`
Expected: `web/src/theme.css` and one of `web/src/index.css` / `web/src/main.css`. Use the `@tailwind`-bearing file as the global (call it `GLOBAL.css`).

- [ ] **Step 2: Add the canvas + motion rules to `GLOBAL.css`** (after the `@tailwind` directives):

```css
html {
  background-color: var(--canvas-base);   /* solid — smoke reads this */
  background-image: var(--canvas);         /* Nocturne gradient */
  background-attachment: fixed;            /* §3.1 — content scrolls over it */
}

/* §3.6 — real motion, opt-out respected. The old stub becomes live. */
@media (prefers-reduced-motion: no-preference) {
  a, button, [role='button'], input, select, textarea,
  .transition-nocturne {
    transition: background-color .15s ease-out, background-image .15s ease-out,
                border-color .15s ease-out, box-shadow .15s ease-out,
                color .15s ease-out, transform .15s ease-out;
  }
}
```

> Do not put a hex in this file. `var(--canvas-base)`/`var(--canvas)` only.

- [ ] **Step 3: Ensure the app root does not paint an opaque bg over the canvas.**

Check the top-level layout (`web/src/shell/AppLayout.tsx` and `web/src/App.tsx`): if the outermost element uses `bg-page`, that's now `#0B0A14` solid (dark) and would hide the gradient. Change the outermost app wrapper to `bg-transparent` (or remove the bg class) so the `html` canvas shows through; inner surfaces (`bg-sidebar`, `bg-card`) are translucent and layer correctly. Leave `bg-page` usages on inner panels alone unless they visibly block the gradient.
